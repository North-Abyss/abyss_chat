1. Abstract
The "Abyss Chat" is a cross-platform, decentralized peer-to-peer (P2P) messaging and
video-calling application built with Flutter. It utilizes WebRTC for true P2P data channels over
the internet and leverages Multicast DNS (mDNS) and local TCP sockets for offline, local-
network discovery. The system provides end-to-end encrypted messaging, high-definition
audio/video calling, and group mesh networking for a seamless and private communication
process.

3. Project Overview – Abyss Chat
3.1 Problem Statement
Traditional messaging applications rely heavily on central servers to route and store
messages. This approach introduces significant privacy concerns, potential data mining, and
single points of failure. Furthermore, users on the same local network cannot communicate
without internet access.
7


3.2 Scope of the Project
To create a completely decentralized P2P application that allows users to send
messages and make audio/video calls without central data storage, working seamlessly both
over the internet (via WebRTC) and locally offline (via mDNS).
3.3 Key Features
• P2P Communication: Direct WebRTC data channels for low-latency messaging and
media.
• Mutual Contacts Only: Strict privacy enforcement, rejecting unknown connections.
• Local Network Discovery: Uses mDNS to discover peers on the same Wi-Fi offline.
• Group Chats & Calls: Supports Full Mesh P2P group chats and video calls.
• Cross-Platform: Runs on Android, iOS, Windows, macOS, Linux, and the Web.
4. System Analysis
4.1 Existing System and Limitations
Existing communication platforms (like WhatsApp or Telegram) have centralized
servers, meaning personal data passes through a third party. They also fail to function offline
even if users share the same local network.
4.2 Proposed System
A decentralized Flutter-based application with an intuitive Material 3 UI, real-time
encrypted WebRTC connections, and offline LAN capabilities.
4.3 Functional Requirements

5. System Design
5.1 System Architecture (Flutter & WebRTC Overview)
Abyss Chat follows a Feature-First Clean Architecture.
• Presentation Layer: Riverpod state providers and Flutter widgets.
• Domain/Data Layer: Controllers handling logic for chat, calling, and contacts.
• Network Layer: WebRTC PeerDart service for signaling/media, LAN Messenger
(mDNS/TCP) for local offline discovery, and Crypto Service for encryption.
5.2 Use Case Diagrams
• User -> Creates Profile, Adds Contacts.
• User -> Initiates WebRTC Call -> Remote Peer.
• User -> Broadcasts mDNS -> Local Peer discovers and connects via TCP.
5.3 Data Flow Diagrams (DFD)
1. Send Message: User Types -> Controller Encrypts via Crypto Service -> Sent via
WebRTC Data Channel or TCP Socket -> Remote Peer Receives -> Decrypts -> UI
Updates.
5.4 Network Topology Diagram
• WebRTC uses a lightweight PeerJS Server strictly for initial SDP/ICE signaling.
• Once signaling completes, a direct P2P mesh network is established between all
participants.

9. Challenges Faced and Solutions
Implemented
• Challenge: Establishing connections without a central server.
• Solution: Used a PeerJS server solely for the initial SDP/ICE handshake; data
channels handle the rest directly.
• Challenge: Storage limitations on the Web platform.
• Solution: Abstracted the storage service to gracefully fall back to
SharedPreferences on Web, while using encrypted file streams on Native.
• Challenge: Local Network Testing failures due to browser mDNS obfuscation.
• Solution: Documented the need to disable
media.peerconnection.ice.obfuscate_host_addresses in Firefox during specific
localhost testing scenarios.
